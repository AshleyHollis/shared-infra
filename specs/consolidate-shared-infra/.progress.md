## Learnings
- HashiCorp recommends `removed` + `import` blocks (Terraform >= 1.7) over `terraform state mv` for cross-state migrations. The `removed`/`import` approach is declarative, PR-reviewable, and maintains audit trail.
- The `moved` block (Terraform >= 1.1) only works within the same state file. It cannot be used for cross-state migration but is useful for post-migration refactoring within shared-infra.
- Azure Blob Storage backend uses blob lease mechanism for state locking. Leases auto-expire after 15 seconds of inactivity. No additional locking config needed.
- When using `terraform state mv` with local state files, the `"serial"` value in the modified source state must be incremented by 1 for the backend to accept the upload. The `removed`+`import` approach handles this automatically.
- Migration must follow dependency order: Resource Group -> DNS Zone -> Key Vault -> ACR -> AKS. Importing a resource before its dependencies causes failures.
- After migration, meal-planner.tfstate may need updating if it references resources via `terraform_remote_state` pointing at the old prod.tfstate location.
- `prevent_destroy = true` lifecycle meta-argument is a critical safety net during migration to protect AKS, ACR, KV from accidental destruction.
- Azure resource IDs for import follow pattern: `/subscriptions/{sub}/resourceGroups/{rg}/providers/{provider}/{type}/{name}`. Can be retrieved via `az resource show --query id -o tsv`.
- Terraform monorepo single root module is viable when resource count stays under ~100. Current scope (AKS, ACR, RG, KV, DNS, networking) is well within this limit.
- Data-only module pattern (wrapping `terraform_remote_state`) is HashiCorp-recommended for cross-repo consumption. Decouples backend config from consumers, enabling future migration without updating every app repo.
- `terraform_remote_state` with azurerm backend exposes full state to anyone with storage read access. Security consideration: only output non-sensitive values.
- OIDC (workload identity federation) is the recommended auth method for GitHub Actions to Azure. Eliminates stored secrets. Requires `id-token: write` permission and federated credentials on Azure AD app registration.
- GitHub Actions path-based triggers (`*.tf`, `modules/**`) are essential for monorepo CI to avoid unnecessary plan/apply runs.
- Azure Blob Storage provides native state locking via blob lease -- no DynamoDB equivalent needed (unlike AWS).
- State backend (storage account for tfstate) is a bootstrap chicken-and-egg problem. Must be created before first `terraform init`. Common approach: Azure CLI one-liner or separate minimal bootstrap config.
- `.terraform.lock.hcl` must be committed to version control to prevent provider version drift between CI and local dev.
- GitHub composite actions cannot access `secrets.*` context directly -- secrets must be passed as inputs via `with:`. Reusable workflows support `secrets: inherit`.
- As of Nov 2025, reusable workflows support up to 10 nesting levels and 50 unique workflows per run (increased from 4/20).
- Git tags apply to entire repo, not individual actions. For monorepo with ~10 actions, unified versioning (single tag for all) is simpler than per-action prefixed tags.
- Renaming a reusable workflow `.yml` file breaks all callers -- GitHub does not follow redirects. Treat filenames as stable API.
- `GITHUB_ENV` variables do NOT propagate between caller and called reusable workflows. Must use explicit inputs/outputs.
- For personal GitHub accounts (non-org), repos must be **public** for cross-repo action sharing. Org accounts can share from private repos with access settings.
- `actionlint` is the standard linter for GitHub Actions YAML files -- should be added to shared-infra CI.
- Composite actions appear as a single collapsed log entry, making debugging harder than inline steps. Add explicit echo debug statements.
- Consumer repos can test shared actions from feature branches using `@branch-name` ref before tagging a release.
- Requirements analysis identified 11 user stories across 6 categories: bootstrap, consolidation, migration, actions, workflows, CI/CD.
- State migration (US-5) is the highest-risk story -- depends on exact config matching and backup procedures. Other stories are lower risk.
- Data-only consumer module (US-6) is a key architectural decision: decouples consumers from backend config, enabling future state backend changes without updating app repos.
- Public repo visibility (DEP-6) is a hard gate for GitHub Actions migration. Must be confirmed before US-8 work begins.
- OIDC app registration reuse vs creation (Unresolved Question 2) affects US-2 scope. Reuse is simpler but creates shared dependency.
- Consumer repo updates (US-7) should be the final step -- depends on both state migration and data-only module being complete.
- Identified 5 unresolved questions that should be answered before implementation begins, particularly repo visibility and OIDC strategy.

- meal-planner uses `data.azurerm_resource_group.shared` and `data.azurerm_key_vault.shared` data sources with hardcoded variable defaults. These are the exact coupling points that the data-only module eliminates.
- Import blocks for role assignments require the full role assignment ID (a GUID), which must be retrieved via `az role assignment list`. Unlike resource IDs, these are not predictable from resource names.
- No `azurerm_dns_zone` or equivalent DNS zone resource exists anywhere in yt-summarizer's Terraform codebase. The only DNS references are AKS `dns_prefix` (part of AKS resource) and a Cloudflare API token variable. DNS zone from requirements (AC-3.1) is deferred -- nothing to migrate.
